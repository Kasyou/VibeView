# 跨项目通用模式与踩坑经验

从所有项目中提炼的通用模式和避坑经验。

---

## 一、文档驱动开发

### 为什么先写文档再写代码

1. **避免方向跑偏** — 写文档时发现的问题，比写代码时发现成本低 10 倍
2. **用户确认更快** — 用户看文档给反馈，比看代码给反馈快得多
3. **Claude 恢复上下文更准** — CLAUDE.md + spec + plan 三位一体，Claude 不需要猜
4. **决策可追溯** — 三个月后还能知道当初为什么选这个方案

### 文档分层

| 层 | 文件 | 用途 | 更新频率 |
|----|------|------|----------|
| L1 | CLAUDE.md | AI 上下文恢复 | 每次状态变化 |
| L2 | spec/*-design.md | 设计决策 | 方向变化时 |
| L3 | plan/*-plan.md | 实施计划 | 按 task 推进 |
| L4 | data/*.json | 运行时数据 | 持续 |

## 二、项目暂停与恢复

### 暂停时必做

1. **更新 CLAUDE.md 的状态标记** — 让人一眼看到当前状态
2. **写清楚阻塞原因** — 是等用户操作？缺资源？还是被动暂停？
3. **写清楚下一步** — "恢复时说'继续做 X'即可"

### 正确示例

```markdown
## Current state
- Status: **PAUSED**
- Blocked: CSDN (微信实名需要用户操作)
- Active platforms: 知乎 + 掘金
- Next on resume: 记录已发布 URL → 查看 Week 1 数据 → 开始 Week 2
```

### 错误示例

```markdown
## Current state
待定
```
（恢复时完全不知道从哪开始）

## 三、JSON 优于数据库

### 什么时候用 JSON 文件

项目数据量 < 10000 条、不需要并发写入、需要版本控制时，JSON 文件比数据库好：

- 零依赖，不需要安装数据库
- 可以用 git 追踪变化
- Claude 可以直接读写，不需要额外工具
- 出问题直接打开文件看

### JSON 文件约定

```json
{
  "version": "1",
  "updated": "2026-05-14",
  "... 实际数据 ..."
}
```

加上 version 和 updated 字段，方便以后迁移和追踪。

## 四、状态标记系统

### 统一的状态标记

| 标记 | 含义 |
|------|------|
| 🟡 | 规划中 — 设计通过了但还没动手 |
| 🔵 | 开发中 — 正在写代码 |
| 🟢 | 已上线/已完成 |
| 🔴 | 已下架/已暂停 |
| ✅ | 售出/达成 |

### 使用场景

- CLAUDE.md 里的项目状态
- 产品列表的状态列
- Plan 里的 task 状态

## 五、命名与约定

### 文件命名

- 设计文档: `YYYY-MM-DD-简短描述-design.md`
- 实施计划: `YYYY-MM-DD-简短描述.md` 或 `YYYY-MM-DD-简短描述-plan.md`
- 日期用 ISO 格式（2026-05-14），方便排序

### 目录命名

- 英文小写 + 连字符: `desktop-organizer`, `image-batch-tool`
- 不要用中文做目录名（终端兼容性）
- 不要用空格（路径处理麻烦）

### Commit 信息

- 英文，简短，动词开头
- `feat:` 新功能
- `fix:` 修复
- `test:` 测试
- `build:` 构建/打包
- `docs:` 文档
- `init:` 初始化

## 六、踩坑记录

### 打包相关的坑

1. **PyInstaller 打包 CustomTkinter** — 需要 `--hidden-import customtkinter`，否则打包后运行报错
2. **中文文件名打包** — `.spec` 文件用 UTF-8 编码，`chcp 65001` 设置控制台编码
3. **打包后体积** — 一个简单 GUI 工具打包后 20-30MB，这是正常的（Python 运行时）

### 多平台发布相关的坑

1. **CSDN 需要微信实名** — 海外用户或没有微信实名的人无法发文章
2. **公众号注册繁琐** — 需要邮箱 + 微信 + 身份证，1-3 个工作日
3. **各平台审核规则不同** — 同一篇文章知乎过了但掘金可能要改

### 开源项目相关的坑

1. **npm publish 需要提前准备** — 包名检查、scope 设置、npm 账号认证
2. **README 是核心营销资产** — 花在 README 上的时间不能省
3. **第一天发布最重要** — 多平台同时发布比分散发布效果好得多

### Claude 开发相关的坑

1. **上下文长了会慢** — 把大任务拆成独立的小模块，每个模块可以在新会话里完成
2. **设计文档不要写太细** — 写到决策层面就够了，实施细节放到 plan 里
3. **CLAUDE.md 要保持更新** — 项目状态变了就要改，否则 Claude 会按旧状态干活
