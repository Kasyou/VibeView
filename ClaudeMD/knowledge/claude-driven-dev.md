# Claude 驱动开发 — 通用方法论

从多个实战项目中提炼的 Claude Code 开发方法论。

---

## 一、项目启动：设计先行

### 标准启动流程

```
用户需求 → 设计文档(design.md) → 实施计划(plan.md) → 分步实现 → 测试验证
```

### 设计文档（spec）应该包含

1. **项目定位** — 做什么，给谁用，一句话说清楚
2. **为什么做** — 市场/需求分析，不做会怎样
3. **核心功能** — P0（必须有）/ P1（加分）/ P2（以后再说）
4. **架构设计** — 核心模块划分，数据流
5. **技术选型** — 用什么技术栈，为什么
6. **关键决策** — 重要的设计取舍和理由
7. **不做什么** — 明确 out of scope

### 实施计划（plan）应该包含

1. **文件清单** — 每个 Task 要创建/修改哪些文件
2. **分步执行** — 每一步做什么，验证标准是什么
3. **测试先行** — 先写测试，确认失败，再写实现，确认通过
4. **Commit 粒度** — 每完成一个功能模块 commit 一次
5. **完成检查** — 最后的 checklist

## 二、CLAUDE.md 设计原则

### CLAUDE.md 是项目的"上下文恢复文件"

每次 Claude 进入项目目录时自动读取。它应该让 Claude 在 30 秒内理解：

1. **这是什么项目** — 一句话
2. **当前状态** — 做到哪了，什么状态
3. **怎么干活** — 工作流程和约定
4. **目录结构** — 关键路径是干什么的
5. **如何恢复** — 接手后第一件事做什么

### 好的 CLAUDE.md 特征

- 简短精炼（< 100 行），不要让 Claude 读太久
- 包含状态标记（🟡规划中/🔵开发中/🟢已完成/🔴暂停）
- 有明确的"如何恢复工作"指引
- 用 `[[项目名]]` 链接关联项目

### 反模式

- 不要把 CLAUDE.md 写成 README — README 是给人看的，CLAUDE.md 是给 AI 看的
- 不要写太细的实现细节 — 那是代码里的事
- 不要写会过时的信息

## 三、项目文件组织

### 推荐结构

```
项目名/
├── CLAUDE.md              ← AI 上下文恢复（必有）
├── [项目代码文件]
├── docs/superpowers/
│   ├── specs/             ← 设计文档（决策和理由）
│   └── plans/             ← 实施计划（分步执行）
├── data/                  ← JSON 数据文件
└── tests/                 ← 测试文件
```

### 文档命名规范

- 设计文档: `YYYY-MM-DD-项目名-design.md`
- 实施计划: `YYYY-MM-DD-项目名-plan.md` 或 `YYYY-MM-DD-项目名.md`

---

## 四、开发节奏

### 推荐节奏

1. **先做设计文档** — 把大的决策先想清楚，避免边做边改方向
2. **拆分独立模块** — 每个模块可以独立完成和测试
3. **TDD 模式更可靠** — 特别是核心引擎逻辑
4. **小步提交** — 每个功能点一个 commit
5. **定期回顾** — 每完成一个 milestone，检查是否偏离设计

### 什么时候改设计

- 实现过程中发现设计有问题的，**先改设计文档再改代码**
- 不要让设计文档变成"摆设"，始终保持它反映真实状态

## 五、与用户协作

### 分工原则

Claude 负责所有"技术实现"工作，用户只做"人类才能做的事"：

**Claude 的活：**
- 代码编写、测试、调试
- 文档撰写、设计规划
- 内容创作、格式化
- 打包、构建

**用户的活：**
- 注册账号、实名认证
- 发布内容到平台
- 收款、发货
- 涉及手机/身份证的操作

### 暂停/恢复机制

项目可能随时暂停。每次暂停前确保：
1. CLAUDE.md 更新了"当前状态"
2. 未完成的工作有明确的下一步
3. 阻塞原因写清楚

恢复时：
1. Claude 读 CLAUDE.md 自动恢复上下文
2. 用户只需说"继续做 X"
